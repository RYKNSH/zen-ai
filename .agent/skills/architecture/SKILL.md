# Architecture Design Skill

## 概要

このスキルは、新機能の設計やシステムアーキテクチャの決定を支援します。
First Principles に基づき、表面的なパターンの適用ではなく、問題の本質を理解した上での設計を行います。

## 発動条件

- 新しい機能やシステムの設計が必要な時
- アーキテクチャの選択に迷っている時
- 既存システムのリファクタリングを計画する時
- 技術選定の判断が必要な時

## 設計プロセス

### Step 1: 問題の本質を理解する

```markdown
## 問題の定義

### 解決すべき問題
[具体的な問題を記述]

### 問題の本質（First Principles）
[表層的な問題ではなく、根本的な課題]

### ステークホルダー
[誰がこの問題の影響を受けるか]

### 成功の定義
[何をもって成功とするか]
```

### Step 2: 理想系を定義する

```markdown
## 理想系

### 制約を無視した理想状態
[技術的・リソース的制約がなければ、どうなるべきか]

### 理想系の特性
- スケーラビリティ:
- 保守性:
- パフォーマンス:
- セキュリティ:
```

### Step 3: 選択肢を列挙する

```markdown
## 設計オプション

### オプション1: [名称]
- 概要:
- メリット:
- デメリット:
- 理想系との距離:

### オプション2: [名称]
- 概要:
- メリット:
- デメリット:
- 理想系との距離:
```

### Step 4: 判断基準を明確にする

```markdown
## 判断基準

| 基準 | 重み | オプション1 | オプション2 |
|------|------|------------|------------|
| 理想系との整合性 | 高 | | |
| 実装コスト | 中 | | |
| 保守性 | 高 | | |
| スケーラビリティ | 中 | | |
| チームの習熟度 | 低 | | |
```

### Step 5: 決定と文書化

```markdown
## 決定

### 選択したオプション
[選択したオプションと理由]

### トレードオフ
[妥協した点と、その理由]

### 将来の拡張パス
[理想系に近づくための次のステップ]
```

## 出力形式

設計が完了したら、以下の形式でADR（Architecture Decision Record）を出力します：

```markdown
## ADR-XXX: [決定のタイトル]

### ステータス
[提案中 | 承認済み]

### コンテキスト
[この決定が必要になった背景]

### 検討した選択肢
[各選択肢の概要]

### 決定
[選択した選択肢と理由]

### 結果
[この決定による影響]

### First Principles 視点
[この決定が根本原因を解決するか、バンドエイドでないかの確認]
```

## チェックリスト

設計を完了する前に確認：

- [ ] 問題の本質を理解したか？
- [ ] 理想系を定義したか？
- [ ] 複数の選択肢を検討したか？
- [ ] バンドエイドになっていないか？
- [ ] ADRに文書化したか？
- [ ] ステークホルダーの合意を得たか？
